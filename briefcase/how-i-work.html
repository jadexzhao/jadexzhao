part 1

ai runtime & the runbook

computers do exactly what you write, never what you meant.

my cs professor's peanut-butter-sandwich drill is the foundation of how i approach frontend architecture and gtm engineering. vague inputs produce unstable outputs. that's the sandwich. below is the operational runbook for co-piloted development.

the runbook: five steps

  1. 01

    constraint formulation

    write prompts with explicit bounds, edge cases, and expected outputs. if the instruction has gaps, the system will fail into those gaps. that's the sandwich.

  2. 02

    ai proposes / human evaluates

    the model suggests architecture or drafts. i evaluate. never swap these roles.

  3. 03

    diff review

    whether inline editing or supervised multi-step agents, every diff gets read and validated before it is accepted. i stay in the diff the whole time.

  4. 04

    system boundaries validation

    check logic, fit, and edge cases against wcag 2.1 and systemic boundaries before anything moves. if an edge case can fail, assume it will.

  5. 05

    human sign-off

    nothing ships without a human reading it and saying yes. major decisions get documented so i can explain them later. drafts co-piloted; accountability entirely mine.

tooling modes

inline editing

reads the repo i'm already in and proposes edits in the diff. i stay in the diff the whole time.

supervised agents

multi-step tool use with human diff review. the assistant chains actions toward a goal. i read every diff before anything lands. supervised means i am the one who says yes.

across both, the rule is the same: AI proposes, i evaluate, and i still own what ships.

when it goes wrong

AI gets it wrong more than people expect. it sounds confident when it's guessing. it fills gaps with plausible-sounding things that aren't true. it optimizes for fluency, not accuracy.

when i catch it, and i do catch it, i don't just fix the output. i go back to the prompt. something in the instruction created room for the wrong answer. that's the sandwich again. the error is a missing step. find the step. fix the instruction. run it again.

what the sandwich taught me

when i build wcag 2.1 compliant components for bloomington nonprofits, or document backend automation scripts in nyc, i look for the unspoken gaps. designing inside heavy institutional parameters like iu rivet or cascade chunks isn't an obstacle. it is a lesson in precision. you outline the assumptions, you test the margins, and you ship the truth.

transparency isn't a policy here. it's an instruction with no missing steps. if you can't see how the work was made, you can't fully trust what it does. that's the whole argument.

part 2

serveit enterprise component matrix

at serveit i approach cascade wcms and rivet as an enterprise component matrix, not a page builder. reverse-engineer the page into modular chunks. keep brand fidelity and accessibility. layout is composition of chunks, not hard-coded glue.

the chunks

text

anchor core narratives without layout noise. zero chrome when the story needs to land first.

callout

flag warnings, mission-critical shifts, or action items readers must not miss.

cta

one primary action per view. no twin competing CTAs fighting for the same click.

stat

metrics for scanning eyes. place high so recruiters and stakeholders get numbers fast.

accordion

secondary specs compressed. keep the first viewport clean; tuck deep docs behind expand.

cascade chunk map · golden paste order

when building a partner page in cascade, i pick chunks in this order so the viewport stays one job at a time:

  1. 1

    top of page

    text block · the hook

    open with the core narrative. no layout chrome.

  2. 2

    above the fold

    stat panel · the scanning metric

    1-3 metrics for scanning eyes. stakeholders get numbers fast.

  3. 3

    mid-page

    callout · the warning

    one warning or action item. never decoration.

  4. 4

    decision point

    cta unit · the singular action

    exactly one primary action per section. zero twin-button competition.

  5. 5

    below the fold

    accordion · the appendix

    secondary specs and long docs. keep the first viewport pristine.

part 4

state isolation protocol

frontend systems fail the same way restaurants do when the floor and the kitchen share one clipboard: conflicting writes, silent overwrites, nobody knows who owns the ticket. i keep a hard global vs local divide so siblings can move without dragging the whole stack.

structural divide

global layer
  • shared env and config that every surface inherits
  • wcag 2.1 aa constants and focus/tap baselines
  • institutional systems: canvas publish shape, clover online ordering (not a fake clover api), rivet design tokens
  • react context and dark mode / theme tokens when the whole app must sync
local layer
  • duck farm coordinates and frame-by-frame refs (raf mutates refs; react only on structure change)
  • map module draft status and publish checklist state
  • etl buffers and validation scratch space
  • isolated listeners (konami, toast timers) that die with the module

three runbook rules

  1. 01

    encapsulation mandate

    keep state as low as possible. do not lift to global unless siblings actually need to sync. if one button needs a flag, that flag lives next to the button.

  2. 02

    one-way data flow

    props down, events up. local modules do not reach into global stores and mutate them side-door style. if theme or auth must change, raise an event the owner understands.

  3. 03

    boundary idempotency

    a local module should run from its props alone. bloomington today, madrid tomorrow: same component quality if the contract is clear. that is transfer quality, not vibes.

the dual-account shape mirrors this without forcing a metaphor. @jadexzhao is the compiler: global architecture, shared tokens, public sandboxes. @matchaxmoxie is the operational loop: local playbooks, canvas checklists, brand floors. same fence line. different jobs inside it.

live example: the virtual duck farm keeps duck coords in local refs during raf; day/night is a small global theme toggle. no setState per frame. honesty check: it is react + typescript + vite. not a d3 canvas demo.

global rules define the fence line. local modules run free inside it. if the fence moves every sprint, nothing ships hot.

part 5

mobile gtm / demo sandboxes

pitch rooms and subway thumbs share one constraint: if the control is hard to hit or the CTA is clipped by chrome, the demo failed the shift. same sandwich rule as cascade chunks. one job per viewport, explicit margins, no silent assumptions.

fat-thumb · 48×48

interactive hits stay ≥48 CSS px. pad the hit box; do not bloat the glyph. primary CTAs live in the bottom ~30% when the layout allows. destructives (clear / wipe) sit out of the casual sweep.

viewport · unit by job

svh for bottom-anchor CTAs and structural nav that must remain reachable with the URL bar expanded. lvh for immersive full-bleed media that should not jitter. dvh for modals, popups, and absolute shells that track live UI chrome.

asset containment

do not ship a desktop hero to a mobile circular crop. prefer picture/srcset, lazy non-LCP media, and capped font requests. demos stay light enough to open on shaky wifi.

advanced mobile rendering for gtm sandboxes

soft 101 ends where the AE opens the sandbox on cellular. this is the bluesheet for pitch-critical motion surfaces.

  1. 01

    compositor motion only

    rAF writes transform / translate3d / scale / rotate / opacity. never per-frame top, left, width, or height. those force layout. cache container size outside the tick (ResizeObserver), then compute px once per write.

  2. 02

    react off the hot path

    no setState inside rAF or high-frequency touch handlers. keep motion state in useRef, mutate the DOM node directly, sync React only on structure changes (spawn, clear, reset). FPS HUD is textContent, not a counter state.

  3. 03

    reserve media geometry

    chart and sandbox stages declare aspect-ratio so the first paint does not shift the CTA row. immersive stages size with lvh; bottom action strips respect svh + safe-area.

  4. 04

    kill touch latency + scroll chaining

    touch-action: manipulation on sandbox triggers, toggles, tabs, and close controls removes double-tap zoom delay. overscroll-behavior: contain on drawers, modals, and scrollable sandbox panels stops elastic scroll from yanking the pitch deck underneath.

live proof: the virtual duck farm keeps spawn in the thumb zone, parks clear-all away from the primary CTA, runs compositor translate3d ducks, and sizes the pond with lvh + aspect-ratio. honesty check still holds: react + typescript + vite. not d3.

part 6

github pages · dual footprint deploy

two public surfaces, two publish shapes. match the workflow to the architecture. do not paste a generic spa playbook onto a static ops site, or a static copy job onto a nested vite demo.

  1. 01

    @jadexzhao · composite artifact

    Actions builds one _site: briefcase HTML at the project root, Vite duck farm nested at /jadexzhao/duck-farm/. that nest is intentional so the pond never clobbers the memo. vite.config base must stay /jadexzhao/duck-farm/ or assets 404 on refresh.

  2. 02

    no spa 404 trap here

    duck farm is a single canvas route. no BrowserRouter, no client path tree. a friendly static 404.html is enough for missing portfolio URLs. the sessionStorage restore rewrite is for apps with real client routes under project Pages. skipping it is correct, not incomplete.

  3. 03

    @matchaxmoxie · static site/

    brand and ops publish the checked-in site/ folder as the Pages artifact. not a Vite app. do not invent base: '/matchaxmoxie/' for that footprint. its 404.html is a branded miss page, not an spa router shim.

  4. 04

    cname isolation

    neither footprint ships a CNAME today (default *.github.io hosts). if a custom domain lands later, keep experiments and sandboxes off the primary hostname so ops and code stay separable. do not point a throwaway demo at the same custom domain as the portfolio without a deliberate split.
